## 1. One-paragraph design

A **property hook** is a class member that looks like a field but computes on read and/or intercepts
writes: `T name { get => expr; set(T v) { stmts } }`. v1 is **virtual-only** (no backing storage of its
own — the motivating `fahrenheit`-from-`celsius` case): a `get` hook is an expression evaluated with
`this` in scope (a read-only computed property); a `set` hook is a block with the assigned value bound
to its parameter, run with `this` in scope (typically writing other `mutable` fields). It transpiles
1:1 to a PHP 8.4 property hook. Reading a hook with no `get` is a compile error; writing one with no
`set` is a compile error (read-only / write-only are both expressible).

## 2. Syntax (locked)

```
class Temp {
    mutable float celsius = ...;          // a normal stored field (set in ctor)

    float fahrenheit {
        get => this.celsius * 9.0 / 5.0 + 32.0;
        set(float v) { this.celsius = (v - 32.0) * 5.0 / 9.0; }
    }
}
```

- `get => <expr>;` — expression body (PHP short form). The expr's type must be assignable to the
  hook's declared type `T`.
- `set(T v) { <stmts> }` — block body; the parameter `v: T` is the assigned value. No return.
- A hook may have **get only** (read-only computed), **set only** (write-only), or **both**.
- A property hook is **virtual**: it declares no storage and takes no initializer; it is NOT `mutable`
  (mutability is a property of storage; a hook is behavior). `get`/`set` reference *other* fields.

## 3. Semantics

- **Read** `o.fahrenheit`: run the `get` hook with `this = o`; its value is the result. No `get` ⇒
  `E-HOOK-NO-GET` (a write-only property can't be read).
- **Write** `o.fahrenheit = e`: bind `v = e`, run the `set` block with `this = o`. No `set` ⇒
  `E-HOOK-NO-SET` (a read-only computed property can't be assigned).
- The `get` expr / `set` block are checked like a 0-arg / 1-arg method body of the class (`this` +
  the class field scope in scope; `v` added for `set`). They may read/write the class's `mutable`
  stored fields (composes with M-mut.6).
- A hook is **not** an instance field: it has no slot in the instance's field map, never appears in a
  constructor's promotion, and is invisible to `clone with`.

## 4. AST

New `ClassMember::Hook` (a distinct variant — structurally different from a stored `Field`: no
storage, no init, carries hook bodies):

```rust
ClassMember::Hook {
    ty: Type,
    name: String,
    get: Option<Expr>,                 // get => expr
    set: Option<(Param, Vec<Stmt>)>,   // set(T v) { stmts }  (Param carries v's name+type)
    span: Span,
}
```

(Adding a `ClassMember` variant breaks the exhaustive matches in loader/checker/interpreter/compiler/
transpiler + the checker rewrite passes — each gets an arm in the same commit, Rust-enforced.)

## 5. Backend lowering

- **Checker**: register hooks per class (`name → (ty, has_get, has_set)`). `o.name` read/write
  resolves a hook before a field; type-check the get expr against `T`, the set body with `v: T`.
  Codes `E-HOOK-NO-GET` / `E-HOOK-NO-SET` / `E-HOOK-TYPE` / `E-HOOK-DUP`.
- **Interpreter**: in `Expr::Member` read, if `name` is a hook of the receiver's class with a `get`,
  run the get expr with `this` bound; in `Stmt::Assign` `Member`, if a hook with a `set`, bind `v` and
  run the block. (Mirrors the static-access detection added in 7a — check the hook table before the
  instance-field path.)
- **Compiler/VM** (the involved part): compile each hook's `get`/`set` as a **synthetic method**
  (`<Class>::<name>$get` arity 0, `<Class>::<name>$set` arity 1) in the function table, dispatched via
  the existing `Op::CallMethod` machinery — **no new `Op`**. `o.name` (read) lowers to a get-method
  call; `o.name = e` lowers to push `e` + a set-method call (result discarded). The compiler's Member
  read/write emit checks the hook table (like 7a's `static_slot`) to choose the method call over
  `GetField`/`SetField`. The get-method's body is the hook expr with an implicit return; the
  set-method's body is the hook block.
- **Transpiler**: emit a PHP 8.4 property hook verbatim:
  ```php
  public float $fahrenheit {
      get => $this->celsius * 9.0/5.0 + 32.0;
      set(float $v) { $this->celsius = ($v - 32.0) * 5.0/9.0; }
  }
  ```

## 6. Differential / examples

- `examples/guide/property-hooks.phg`: a `celsius`/`fahrenheit` temperature class (get computes, set
  back-converts), read + write through both, byte-identical on run/runvm/real PHP. Plus a validation
  example (a `set` clamps a stored field).
- Differential `mutation_property_hooks_agrees`: read computed, write-through-set observed on the
  backing field, read-only hook write rejected (compile error, not a runtime case).

## 7. Deferred (KNOWN_ISSUES)

Backed hooks (a hook with its own storage + the PHP `$field` magic / `$this->name` self-reference);
hooks on `static` fields; hooks in interfaces; abstract/overridable hooks. v1 is virtual-only.

## 8. Open question to resolve at build

Does the VM's synthetic-method dispatch interact with method **name collision** (a real method named
`fahrenheit$get`)? Mangle with a character illegal in a Phorge identifier (e.g. the `$`), so a
user method can never collide — confirm the name-pool keying tolerates it.
