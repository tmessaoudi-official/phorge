# M8.5 S3 — `.d.phg` declaration files + foreign-exception catch — design

Status: **design-locked** (2026-06-28). Third item of marathon #3 ("all of them in order").
Builds on the M8.5 interop core (`docs/specs/2026-06-28-m8.5-interop-design.md`, S1 `declare function`
+ S2 `declare class`). Spine rule unchanged: foreign symbols are **PHP-target-only**, quarantined from
the `run ≡ runvm` byte-identity oracle (`E-FOREIGN-RUNTIME`); validated by transpile → real PHP golden.

## Two sub-slices

### S3a — foreign-exception `catch`

**Goal.** Let a Phorj program `catch` an exception thrown by foreign PHP code (e.g. PHP's
`intdiv(10, 0)` throws `\DivisionByZeroError`).

**The model.** A foreign PHP exception is described by a `declare class` that **implements the built-in
`Error` marker** — the same marker Phorj's own `throws`/`catch`/`Result` machinery is built on:

```
declare class DivisionByZeroError implements Error {
  function getMessage() -> string;
}
declare function intdiv(int a, int b) -> int;

function main(): void {
  try {
    int q = intdiv(10, 0);
    Console.println("q={q}");
  } catch (DivisionByZeroError e) {
    Console.println("caught: {e.getMessage()}");
  }
}
```

**The two gaps (both bounded).**
1. **Parser** (`parse_declare_class`) hardcodes `extends: []` / `implements: []`. Fix: parse an optional
   `extends <names>` and/or `implements <names>` clause after the foreign class name (reusing the
   ordinary class-header helpers). A foreign exception writes `implements Error`; a foreign class may
   also `extends` another foreign class. The names describe the *PHP* hierarchy — purely a type-checker
   input, never a body.
2. **Transpiler** `php_catch_type` is foreign-blind: a bare `Type::Named("DivisionByZeroError")` would
   emit a bare `DivisionByZeroError`, which inside a `namespace Main { … }` block resolves to
   `\Main\DivisionByZeroError` (wrong). Fix: make catch-type emission **foreign-aware** — a foreign
   class name emits the global form `\Name` (consistent with S2's `new \Name` / `\Name::s`). Threaded
   through `self.foreign_classes`.

The checker already registers a foreign class in the shared `class_implements` oracle (S2), so
`implements Error` makes `is_subtype(name, "Error")` true → the `catch` type-checks (`E-CATCH-TYPE`
otherwise) and `instanceof`/member-call resolution work with no checker change. `run`/`runvm` already
refuse the whole program (`E-FOREIGN-RUNTIME`) — foreign catch is a transpile-only behavior, so there
is **no backend execution path and no new `Op`/`Value`**.

**Note on the `Error` marker vs PHP `\Throwable`/`\Error`.** Phorj's bare `Error` marker transpiles a
*catch* to `\Exception` (M-faults 2b — Phorj's own exceptions `extends \Exception`). A **named**
foreign exception is caught by **its own** name (`\DivisionByZeroError`), NOT the `Error`→`\Exception`
mapping — so a foreign `\Error`-family class (which is a `\Throwable` but not an `\Exception`) is caught
correctly. The example deliberately uses `\DivisionByZeroError` (an `\Error`-family class) to prove this.

### S3b — `.d.phg` ambient declaration files

**Goal.** Lift the bundle of `declare`s out of every consumer file into a shared `*.d.phg` file (the
`.d.ts` analog), so a project declares its foreign surface once.

**Discovery (ambient-within-project, no new syntax).** A file whose name ends with `.d.phg` is a
**declaration file**. Its presence in the project source tree *is* the opt-in (exactly like `.d.ts` in a
TS compilation) — no `import` statement, consistent with both "explicit" (you added the file) and the
fact that foreign PHP symbols are globally addressable (`\strlen`) with no package to qualify them.

A `.d.phg` file:
- requires **no `package`** declaration (and **must not** carry one — `E-DECL-PACKAGE`: a declaration
  file describes global foreign PHP, which has no package);
- must contain **only `declare` (foreign) items** — `E-DECL-NONFOREIGN` on any non-foreign item;
- is **exempt** from folder=path (`E-PKG-PATH`) and the public-surface rule (foreign items already are);
- its foreign items are merged **ambiently** into the unified program — available to every package.

**The three loader gaps (all bounded).**
1. `collect_phg` / `walk` pick up `foo.d.phg` as an ordinary `.phg` (extension is `phg`). Fix: split the
   walk — ordinary collection **excludes** `*.d.phg`; a sibling collector gathers `*.d.phg`.
2. `resolve_item` (the cross-package name-mangle pass) would mangle a foreign symbol
   (`\strlen` → `\Pkg\strlen`). Fix: `resolve_item` **early-returns foreign items unmangled** (also
   latent-correct for an inline `declare` inside a library package).
3. The merged foreign items are validated (no package, all-foreign) at load, then concatenated ahead of
   the regular files. The single `E-NO-PACKAGE` check runs on the merged program (which has the entry's
   `package Main`), so package-free decl files don't trip it.

**Testing without breaking the spine.** A `.d.phg`-using project is foreign → `run`/`runvm` refuse it,
so it **cannot** be byte-identity-gated by the project-aware differential harness. Mitigation: the
example project lives under `examples/interop/` and the differential project collector skips the
`interop` subtree (mirroring the existing single-file skip). It is validated instead by an extended
`tests/interop.rs`: a project directory loads via `loader::load`, `run`/`runvm` refuse it
(`E-FOREIGN-RUNTIME`), and `transpile` → real PHP matches the committed golden.

## Verification

- S3a: `examples/interop/exceptions.phg` (+ `.out` golden) — auto-covered by `tests/interop.rs`.
- S3b: `examples/interop/withdecls/` project (a `*.d.phg` + a `src/main.phg`) + `.out` golden; new

- Whole gate: `cargo test --workspace` with `PHORJ_PHP=/stack/tools/phpbrew/php/php-8.5.7/bin/php
  PHORJ_REQUIRE_PHP=1`; `cargo clippy --all-targets -- -D warnings`; `cargo fmt --check`. Pure-Phorj
  suites unchanged (spine untouched).

## Deferred (KNOWN_ISSUES)

`.d.phg` *generation* from PHP source; Composer-package declaration bundling; namespaced foreign symbols
beyond a single leading `\`; foreign generic PHP types; running foreign programs on the Rust backends
(needs a PHP FFI). A foreign `throw new \Name(...)` is constructible (S2 `new \Name`) but idiomatic
interop catches foreign exceptions rather than raising them.
